# VRT Space Agency — Codex Master Instructions

## Purpose

This document is the operating guide for Codex while restructuring and improving the VRT Space Agency project.

The goal is to transform the current build into a clean, simple, high-converting SaaS product focused on:

* **Audit**
* **SEO**
* **AEO**

The site must feel premium, clear, and trustworthy. It should not feel like a portfolio, an agency brochure, or a complicated dashboard. It should feel like a focused product that helps users understand what is wrong with their website, what to fix, and how to improve over time.

---

## Product Vision

VRT Space Agency is a SaaS platform that helps users improve website visibility through three connected layers:

1. **Audit** — scans the live website and identifies immediate issues and opportunities.
2. **SEO** — analyzes missing patterns, competitor gaps, backlinks, and strategic improvements.
3. **AEO** — uses the audit and SEO findings to assess answer-readiness and AI visibility.

The product should create a loop:

**Audit → SEO → AEO → Workspace → Rerun → Progress → Retention**

This loop is the heart of the product.

---

## Primary Business Goals

1. Convert visitors into signups and paying users.
2. Make the first 3–5 minutes on the site feel valuable and intelligent.
3. Encourage repeated usage through workspaces, reruns, and visible progress.
4. Use credits naturally so users keep returning.
5. Create a product that can support affiliate growth later.

---

## What Must Be Kept Simple

The public-facing experience must stay focused on only three things:

* SEO
* Audit
* AEO

Everything else should be hidden, deferred, or placed behind future expansion.

Do not overbuild. Do not add complexity just because it is possible. Every screen must help the user understand value faster.

---

## Core Product Behavior

### Audit

The audit is the first wow moment.
It should:

* scan the live website
* use Google PageSpeed Insights API
* use SERP data through SerpAPI
* surface performance blockers, indexing issues, mobile issues, basic visibility gaps, and obvious technical concerns
* present results in a simple, decisive way

The audit output should not feel like a long report. It should feel like a quick diagnosis with a clear conclusion.

### SEO

The SEO layer should be strategic and comparative.
It should:

* identify missing content and missing patterns
* compare with competitors
* inspect backlink gaps
* reveal keyword/topic opportunities
* show what should be built or improved

The SEO output should feel like a practical roadmap.

### AEO

AEO should depend on the Audit and SEO findings.
It should:

* measure answer-readiness
* check entity coverage
* check topic coverage
* check structured data readiness
* check content suitability for AI search and answer engines
* reveal what is needed to improve AI visibility

AEO should feel like the premium intelligence layer.

---

## Workspace Philosophy

The workspace is not just a folder. It is the user’s active growth space.

Each workspace should allow the user to:

* create and manage separate projects
* store audit history
* store SEO history
* store AEO history
* rerun analyses after fixes
* compare before and after results
* see progress over time
* consume credits naturally through meaningful actions

The workspace should make users feel that their website is being improved over time, not just analyzed once.

---

## Credit System Philosophy

Credits must feel like access to progress, not a penalty.

Users should spend credits on meaningful actions such as:

* running an audit
* running SEO analysis
* running AEO analysis
* rerunning after changes
* deeper competitive checks
* deeper visibility checks

The goal is for credits to feel tied to improvement and discovery.

---

## Conversion Goals

The product must support the following conversion logic:

### Visitor stage

* User lands on the homepage.
* User understands what the product does in seconds.
* User sees one main CTA.

### Trial stage

* User enters a website URL.
* User gets value quickly.
* User sees meaningful issues and opportunities.

### Upgrade stage

* User runs out of credits.
* User sees what additional insight or progress they can unlock.
* User upgrades because they want continuation, not because of pressure.

### Retention stage

* User returns to rerun the audit/SEO/AEO after fixes.
* The platform shows progress clearly.
* The workspace becomes part of their operating routine.

## Core UX Principles

### 1. Clarity first

Every page must answer:

* What is this?
* Why does it matter?
* What should I do next?

### 2. One primary action

Each screen should have one dominant next step.
Avoid too many equal-weight buttons.

### 3. Fast value

The product must show meaningful insight in the first 3–5 minutes.

### 4. Calm confidence

The interface should feel premium, organized, and intelligent.

### 5. Visible progress

Users should always feel like they are moving forward.

### 6. No clutter

Avoid stacking too many cards, badges, panels, or metrics.

### 7. No jargon walls

Use plain language where possible. Reserve technical terms for the right context and explain them naturally.

---

## UI Design Direction

The UI should feel:

* modern
* minimal
* premium
* sharp
* readable
* conversion-focused

### Visual hierarchy

Use strong hierarchy:

* large headline
* short supporting copy
* clear CTA
* supporting sections beneath

### Spacing

Give the design room to breathe.
Use generous spacing instead of stuffing content together.

### Layout

Favor simple, structured layouts:

* hero
* value section
* product sections
* process section
* trust section
* CTA

### Motion

Use subtle motion only when it helps the user understand progress.
Avoid decorative motion that distracts.

### Mobile-first behavior

The product must work exceptionally well on mobile.
The mobile layout should preserve:

* clarity
* speed
* readability
* easy tapping

---

## Best Conversion Patterns to Use

### Hero section

The hero must immediately communicate:

* what the product does
* who it is for
* why it is useful
* what the next step is

It should include:

* one strong headline
* one short supporting paragraph
* one primary CTA
* one optional secondary CTA

### Simple promise

The headline should promise a result, not describe a feature list.

### Visual proof

Use one simple visual, sample output, score preview, or concept illustration that reinforces the promise.

### Problem → insight → action flow

Every important section should follow:

* problem
* what the system found
* what the user should do next

### Controlled reveal

Show enough to create trust, then unlock deeper value after the user is engaged.

### Repetition without noise

Repeat the main message in different forms across the page, but do not flood the user with copy.

### One CTA family

Use the same action language across the site where possible.
Examples:

* Start Audit
* Analyze Website
* Create Workspace
* Rerun Analysis

---

## Messaging Rules

### Use plain language

Say what the user gets.
Do not hide the benefit behind abstract terms.

### Speak in outcomes

Emphasize:

* visibility
* clarity
* missed opportunities
* progress
* growth
* AI answer readiness

### Avoid fake hype

Do not overpromise. The system should feel powerful because it is useful, not because it makes exaggerated claims.

### Make the user feel understood

The copy should make the visitor think:

* “This understands my website.”
* “This shows me what is missing.”
* “This gives me a path forward.”

## Audit UX Requirements

The audit experience should:

* start quickly
* show active scanning behavior
* reveal findings in a structured way
* end with a clear conclusion

Suggested output structure:

* visibility score
* performance score
* critical issues
* quick wins
* recommended next action

The audit should not feel long or exhausting.

---

## SEO UX Requirements

The SEO experience should:

* compare the user against competitors
* show missing patterns
* show backlink or authority gaps
* show content or keyword opportunities
* prioritize actions by impact

Suggested output structure:

* competitor gap summary
* missing content patterns
* backlink/authority observations
* priority actions
* expected impact notes

---

## AEO UX Requirements

The AEO experience should:

* depend on the audit and SEO data
* show answer-readiness in a clear way
* expose entity and topic gaps
* explain how the site can become AI-search friendly

Suggested output structure:

* answer-readiness score
* entity coverage
* topical coverage
* structured data readiness
* AI visibility opportunities
* top fixes

---

## Workspace UX Requirements

Each workspace should feel like a project dashboard with a growth history.
It should include:

* project name
* website URL
* audit history
* SEO history
* AEO history
* rerun controls
* comparison history
* credit usage
* progress indicators

The workspace should make it easy to come back, make changes, and measure improvement.

---

## Rerun Strategy

Reruns are a core retention mechanic.

After fixes, the user should be able to rerun:

* audit
* SEO
* AEO

The product should clearly show:

* what improved
* what remains
* what changed since the last run
* what is now unlocked

This makes the product feel alive.

---

## Trust and Credibility

The product should build trust through:

* clear structure
* honest copy
* visible methodology
* sensible outputs
* progress tracking
* professional design

Do not use fake testimonials or inflated claims.

If proof sections are needed, they should be simple and believable.

---

## Future Affiliate Readiness

Even if affiliate systems are not fully built yet, the product should eventually support:

* referral growth
* partner landing pages
* easy-to-explain value
* strong conversion pages
* recurring usage and retention

Design the product so it is easy to recommend.

---

## What Codex Must Not Do

* Do not reintroduce unnecessary modules into the public UI.
* Do not make the product feel like a generic agency site.
* Do not make it look like a cluttered admin dashboard.
* Do not add unnecessary features that dilute the core product.
* Do not sacrifice clarity for novelty.
* Do not break existing working behavior unless a change is necessary for conversion or structure.

---

## What Codex Should Do

1. Inspect the current implementation.
2. Identify the biggest conversion blockers.
3. Restructure the public-facing experience around clarity and value.
4. Keep only Audit, SEO, and AEO visible in the core product.
5. Improve homepage flow and messaging.
6. Strengthen the workspace and rerun experience.
7. Make the UI feel premium, simple, and trusted.
8. Preserve and improve existing functionality rather than replacing everything unnecessarily.

---
